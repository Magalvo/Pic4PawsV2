## 3. Deviations & Scope Creep

### D1 — Client-layer sanitization tests do not assert the `bearer` pattern

**Severity**: Low  
**Files**: `tests/client/pet-draft-load-client.test.ts`, `tests/client/pet-feed-client.test.ts`

The SDD convention (`docs/canonical/sdd.md`) requires that sanitization tests assert BOTH `service-role` AND `bearer` pattern substrings are absent from the serialized failed state. The two client tests deviate:

`tests/client/pet-draft-load-client.test.ts` (introduced in PR #127):
```js
expect(JSON.stringify(result)).not.toContain('service-role-secret');  // specific literal, not pattern
expect(JSON.stringify(result)).not.toContain('user-token');            // no 'bearer' pattern check
```

`tests/client/pet-feed-client.test.ts` (pre-existing file, touched in PR #128):
```js
expect(serialized).not.toContain('service-role-secret');  // specific literal, not pattern
// no 'bearer' assertion at all; no bearer marker in the test input
```

**Fix**: In `tests/client/pet-draft-load-client.test.ts`, replace `service-role-secret` with `service-role` and add `expect(JSON.stringify(result)).not.toContain('bearer ')`. In `tests/client/pet-feed-client.test.ts`, add `'bearer token-value'` to the input reasons and assert `not.toContain('bearer ')`.

The web and mobile `pet-draft-load` tests are correct — they have a second test that explicitly checks both `service-role` and `bearer ` patterns.

### D2 — SHELTER-PETS work item status fields not updated after merge
