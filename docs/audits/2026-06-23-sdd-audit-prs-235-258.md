## 3. Deviations and Scope Creep

**N-01 — Local helper duplication in `push-token.ts`**

`apps/workers/src/push-token.ts:29–38` defines private copies of `jsonResponse` and
`extractBearerToken` instead of importing from `apps/workers/src/routes/shared.ts`.
This extends the F-13 pattern from `donation-manual.ts` to a third handler. If
`shared.ts` changes its Bearer-parsing logic, `push-token.ts` and `donation-manual.ts`
silently diverge. Straightforward fix: delete local copies, import from `'./routes/shared'`.

**N-02 — Auth ladder order deviation in `push-token.ts`**

`PUSH-TOKEN-WORKER-001` spec contract: `405 → 501 (no authenticator) → 401 → 501 (no repo)`.
Implementation order: `405 → 401 (no bearer token extracted) → 501 (no authenticator) → 401
(bad/expired token) → 501 (no repo)`. The bearer token presence check runs before the
authenticator-configured check. Functionally equivalent and marginally more secure (does not
reveal auth capability when no bearer is provided), but deviates from the stated contract.
Not a bug; document or align in a future hygiene PR.

No scope creep identified in any of the in-scope PRs. `DONATE-TIER-PLAN-001` (PR #241) is
a docs-only PR that correctly precedes the implementation PRs.

One architectural note: `DONATE-CONFIG-WORKER-001` completion notes mention the Supabase
implementation uses two sequential queries (upsert + UPDATE shelters) rather than a single
RPC transaction. The spec says "same DB operation" or recommends atomic execution. Two
sequential queries are not truly atomic — a failure between them leaves the shelter in
`not_configured` (the safe failure mode). The notes acknowledge this is safe. However, if
the UPDATE to `shelters.payment_account_status` fails silently (error not propagated), the
shelter config row exists but the shelter remains un-activatable. The upsert can succeed
but the shelter activate-step can fail if there is a connection issue between the two calls.
This is a P3 observation — the failure mode is visible at the DB level and the UX handles
it by retrying save.

## 4. Actionable Next Steps

**P1-1 (Critical, carried F-01) — Fix false HTTP 200 on concurrent donation state change**

`apps/workers/src/donation-manual-supabase.ts` lines 85, 99, 118: `submitReceipt`,
`approveDonation`, and `rejectDonation` all call `.update()` without `.select('id')`.
Supabase returns `{ data: null, error: null }` regardless of 0-row or N-row match.
A concurrent PATCH passes the in-process status guard but the WHERE clause matches 0 rows;
the caller receives HTTP 200 and `notifyDonationPaid` may fire for a phantom approval.

Fix: chain `.select('id')` on each update. If `result.data?.length === 0`, return
`donation_wrong_state` (409). Precedent: `.maybeSingle()` pattern in `getDonationForAction`.

**P1-2 (Critical, carried F-02) — Wire real file picker for receipt upload**

`apps/mobile/app/doacoes/[donationId]/comprovativo.tsx:84` constructs a hardcoded
mock `MobileDonationReceiptFileInput` (`uri: 'file:///data/placeholder/receipt.jpg'`,
`size: 0`). `expo-image-picker` and `expo-document-picker` are absent from
`apps/mobile/package.json`. Every receipt upload sends this placeholder — the feature
is non-functional for real donors.

Fix: Add `expo-image-picker` to `apps/mobile/package.json`. In `handleSelectAndUpload`,
replace the mock object with `await ImagePicker.launchImageLibraryAsync(...)` and map
the result to `MobileDonationReceiptFileInput`. Handle permission denial and
null-result (user cancelled) gracefully. Fix F-03 at the same time.

**P2-1 (High, carried F-03) — Fix wrong `purpose: 'identity_document'` on receipt uploads**

`apps/mobile/src/donation-receipt.ts:209` and `apps/web/src/donation-receipt.ts:195` use
`purpose: 'identity_document'` for receipt media uploads. Receipts are filed under the
identity-document storage path. Fix: replace with `'donation_receipt'` or the purpose
value defined for this upload category. Confirm the media-upload worker accepts this value.
Must be resolved before F-02 and F-08 can be fully closed.

**P2-2 (High, carried F-04) — Add `useRef` to `pagamento.tsx` `handleSubmit`**

`apps/mobile/app/abrigos/[shelterId]/pagamento.tsx:63–88`: `handleSubmit` creates fresh
`createSavePaymentConfigClient`, `createLoadPaymentConfigClient`, and
`createMobileShelterPaymentConfigUi` instances on every invocation. Two rapid presses
launch two independent `saveConfig` calls with no concurrency guard.

Fix: add `useRef<ReturnType<typeof createMobileShelterPaymentConfigUi> | null>(null)` at
component top; store the UI in `useEffect`; reuse it in `handleSubmit`. Change the guard
from `if (submitting || !viewModel) return` to `if (submitting || !viewModel || !uiRef.current) return`.
Precedent: `comprovativo.tsx` `uiRef` pattern.

**P2-3 (High, carried F-05 / F-07) — Wrap Alert onPress callbacks in try/catch; add null guard**

`apps/mobile/app/doacoes/[donationId]/comprovativo.tsx:73` and
`apps/mobile/app/abrigos/[shelterId]/doacoes/[donationId].tsx` (handleApprove/handleReject):
Alert's async `onPress` callbacks are not wrapped in try/catch; unhandled rejections leave
the screen stuck in an intermediate state. Additionally, the `uiRef.current!` non-null
assertion inside the async callback can crash on component unmount between Alert display
and user tap.

Fix: wrap each async `onPress` body in `try { ... } catch { setViewModel({ state: 'failed', ... }) }`.
Add `if (!uiRef.current) return;` at the top of each async `onPress` callback before the assertion.

**P3-1 (Medium, carried F-06) — Use `mobileSupabaseClient` singleton in donation review screen**

`apps/mobile/app/abrigos/[shelterId]/doacoes/[donationId].tsx:20–21,31`: `createClient()`
from `@supabase/supabase-js` is called directly inside `useEffect`, creating a new Supabase
client with a fresh auth state machine and realtime WebSocket on every mount.

Fix: replace the raw `createClient` import + call with `mobileSupabaseClient` (singleton
from `apps/mobile/src/supabase.ts`). Precedent: all other mobile screens use the singleton.

**P3-2 (Medium, carried F-08) — Scope `verifyMediaOwnership` by purpose**

`apps/workers/src/donation-manual-supabase.ts:67`: `verifyMediaOwnership` filters only
by `created_by_user_id`. Any media asset the donor previously uploaded (pet photo, etc.)
passes the check and can be submitted as a receipt. Fix after P2-1 (F-03): add
`.eq('purpose', 'donation_receipt')` (or the chosen value) to the ownership check query.

**P3-3 (Medium, carried F-09) — Evaluate `now()` at write-time, not dispatch-time**

`apps/workers/src/routes/donations.ts:45`: `now: dependencies.now?.() ?? new Date().toISOString()`
captures the timestamp before auth, validation, and async work. Pass the factory instead:
`now: dependencies.now ?? (() => new Date().toISOString())` and evaluate it inside
`handleReviewDonationRequest` immediately before the DB write call.
