## 1. Spec Compliance Score: 7/10

The remediation wave is real and high-quality: all fourteen findings from the 2026-06-23
audit (F-01 through F-14) and N-01 are genuinely closed in the current code, with dedicated
regression tests for the two Critical bugs (F-01 false-200 concurrency race; F-02 hardcoded
receipt placeholder). Auth invariants, server-clock timestamps, soft-delete guards, and
sanitization tests are all correct. PET-FEED-FILTERS-001 and WORKER-DISPATCH-MODULAR-001
remain spec-compliant.

Three points deducted:

- **−2 (Critical regression introduced by the F-03 fix)**: The F-03 remediation (PR #262)
  changed both receipt-upload boundaries to send `purpose: 'donation_receipt'`, and extended
  the `MediaUploadPurpose` domain type to include it — but the **Worker's runtime upload-purpose
  allowlist (`mediaUploadPurposes` in `apps/workers/src/media-upload.ts`) was never updated**.
  Every receipt upload now fails at the Worker with `400 invalid_upload_request /
  ['invalid_purpose']`. The manual donation receipt flow is non-functional end-to-end —
  the exact failure class F-02/F-03 set out to eliminate. See P1-1.
- **−1 (Test gap that masked the regression + a doc over-claim)**: No test exercises the
  Worker's `mediaUploadPurposes` allowlist with `donation_receipt`; the only tests touching
  that value are the two UI tests, which mock the media-upload client and never reach Worker
  validation. The checkpoint also claims N-02 was "remediated" when the auth-ladder order was
  not changed. See P2-1 and N-03.

This score is for the *current state of `main`*, not the effort: the remediation work itself
would merit a higher mark, but it shipped a Critical functional regression that the gate
suite does not catch.

## 3. Deviations & Scope Creep

**P1-1 (see §4) is the dominant deviation**: a Critical functional regression introduced as
a side-effect of a remediation. The boundary–Worker contract for upload purpose drifted: the
domain type and both UI boundaries advanced to `donation_receipt`, but the Worker's runtime
allowlist did not. This is the canonical "type allows it, runtime rejects it" gap, invisible
to typecheck.

**N-03 — N-02 over-claimed as remediated.** The 2026-06-23 (235-258) audit classified N-02 as
"Not a bug; document or align." The checkpoint `2026-06-23-donation-manual-complete.md` lists
"PR #266 — F-02 + N-02" as closed, but `push-token.ts`'s auth ladder is byte-for-byte the same
ordering the audit flagged (bearer-presence 401 before authenticator-configured 501). Either
the spec contract in `PUSH-TOKEN-WORKER-001` should be updated to ratify the implemented order,
or the ladder reordered — but the remediation log should not record N-02 as fixed when no code
or spec changed. Low severity.

**N-04 — Partial helper consolidation leaves two parallel modules.** The F-13/N-01 fix routed
`donation-manual.ts` and `push-token.ts` onto the new `apps/workers/src/http.ts`. But
`apps/workers/src/routes/shared.ts` still defines its own `jsonResponse` and
`parseAuthorizationBearer`, and the `routes/*.ts` dispatch modules import from `shared.ts`.
The codebase now has two near-identical helper homes. Both are trivial `Response.json` /
Bearer-parse wrappers so divergence risk is low, but the "single canonical home" intent stated
in the checkpoint is only half-achieved. Low severity / informational.

**N-05 — F-07 review-screen Alert callbacks lack try/catch.** `handleApprove`/`handleReject`
in the mobile review screen guard `uiRef.current` (F-07 closed) but do not wrap the async
`onPress` body in try/catch, unlike `comprovativo.tsx`. The review boundary
(`createMobileDonationReviewUi`) is designed to return a `failed` view-model rather than throw,
so a stuck-UI freeze is unlikely; this is defence-in-depth, not a live bug. Low severity.

No scope creep in the feature PRs. The remediation PRs stayed within their findings.

## 4. Actionable Next Steps

**P1-1 (Critical) — Add `donation_receipt` to the Worker upload-purpose allowlist**

`apps/workers/src/media-upload.ts` line ~102, the `mediaUploadPurposes: MediaUploadPurpose[]`
array lists `pet_public_image`, `shelter_public_image`, `adoption_document`,
`identity_document`, `medical_record` — but not `donation_receipt`. `parseWorkerMediaUploadPayload`
(the single validation path for `createWorkerMediaUploadIntent`) rejects any other value with
`reasons: ['invalid_purpose']`, returning `400 invalid_upload_request`. Because both
`apps/web/src/donation-receipt.ts:193` and `apps/mobile/src/donation-receipt.ts:210` now send
`purpose: 'donation_receipt'`, every real receipt upload fails — the manual donation receipt
flow is non-functional end-to-end, and the F-08 `verifyMediaOwnership` purpose filter can never
match because no receipt asset can be persisted.

Fix: add `'donation_receipt'` to `mediaUploadPurposes`. Confirm the resulting visibility is
correct — `visibilityForPurpose` in `packages/domain/src/media-policy.ts` already treats
`donation_receipt` as private (good; receipts must not be public). Verify any storage-bucket
or path routing keyed on purpose has a `donation_receipt` destination. This must land before
any user-facing receipt upload.

**P2-1 (High) — Close the test gap that hid P1-1**

There is no test asserting the Worker accepts `donation_receipt` as an upload purpose. The
only tests referencing the value (`tests/web/donation-receipt-ui.test.ts`,
`tests/mobile/donation-receipt-ui.test.ts`) mock the media-upload client and never reach
`media-upload.ts` validation. Add a case to the Worker media-upload test suite
(`tests/workers/media-upload*.test.ts`) that submits `purpose: 'donation_receipt'` and asserts
`ok: true` (not `invalid_purpose`). Ideally, add a single assertion that the Worker's
`mediaUploadPurposes` allowlist is a superset of the domain `MediaUploadPurpose` union, so a
future purpose added to the type but not the allowlist fails CI rather than production.

**P2-2 (Low) — Reconcile N-02 status (doc vs code)**

Either reorder the `push-token.ts` auth ladder to `405 → 501 (no authenticator) → 401` to
match the `PUSH-TOKEN-WORKER-001` contract, or update that work item's Contract section to
ratify the implemented order (bearer-presence check first). Then correct the checkpoint
`2026-06-23-donation-manual-complete.md`, which records N-02 as remediated although no code
or spec changed. File: `apps/workers/src/push-token.ts`;
`docs/work-items/` PUSH-TOKEN work item; `docs/checkpoints/2026-06-23-donation-manual-complete.md`.

**P3-1 (Low) — Finish the helper consolidation**

Make `apps/workers/src/routes/shared.ts` re-export `jsonResponse` (and a single Bearer-parse
helper) from `apps/workers/src/http.ts` instead of redefining them, so there is one canonical
implementation. Optionally fold `parseAuthorizationBearer` and `extractBearerToken` into one
name. Files: `apps/workers/src/routes/shared.ts`, `apps/workers/src/http.ts`.

**P3-2 (Low) — Defensive try/catch in review-screen Alert callbacks**

For parity with `comprovativo.tsx`, wrap the async `onPress` bodies of `handleApprove` /
`handleReject` in `apps/mobile/app/abrigos/[shelterId]/doacoes/[donationId].tsx` in try/catch
that sets a `failed` view-model. Defence-in-depth; the boundary currently absorbs errors.

**P3-3 (Informational) — Update agent-resume after P1-1 fix**

`docs/agent-resume.md` §4 states the manual-donation slice is "complete + audit-remediated".
Until P1-1 lands, the receipt upload path does not work in production; the resume should not
