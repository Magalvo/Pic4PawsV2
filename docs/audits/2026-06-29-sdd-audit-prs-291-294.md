## Actionable Next Steps

### Advisory (non-blocking)

**1. Harden `DonationRepository` interface type safety**

File: `apps/workers/src/donation.ts:75-76`

Remove `?` from `setProviderPaymentId` and `failDonation`. The Supabase implementation
(`apps/workers/src/donation-supabase.ts:159,170`) already provides both, so removing the
optionality from the interface will not break existing code. It will:

- Eliminate the runtime `if (!donationRepository.setProviderPaymentId)` guard, which
  becomes unreachable once the type is non-optional.
- Enforce at compile time that all future `DonationRepository` implementations supply
  both methods.
- Align the code with the work item goal ("always call `failDonation` before returning").

Note: the test at line 494 constructs a repository without `setProviderPaymentId`
intentionally to test the guard path. If the interface becomes non-optional, that test
must be refactored (e.g., via `as unknown as DonationRepository` cast or by casting the
partial object). The behavior under test (502 when persistence is unavailable) should
still be covered after the type change.

This is low-risk housekeeping; it does not need a dedicated SDD work item unless the team
prefers to track it formally.
