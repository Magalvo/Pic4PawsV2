## 3. Deviations & Scope Creep

**No scope creep detected.** All new files correspond to stated work items.

**Design deviation — web `guardar` save trigger on file-select (P4 observation):**
The web `guardar/page.tsx` implements two save paths: `handleFileChange` triggers an
immediate `saveDraft` when a file is selected, while `handleSave` saves without a new
file. This conflates file selection with form submission — a user who accidentally selects
a file will trigger a save immediately without a confirm step. This is a UX concern
rather than an SDD violation, but worth noting.

**Dual-boundary pattern on `guardar` screens (P5 observation — acceptable):**
Using `createMobilePetDraftUi` for loading and `createMobilePetDraftSaveFlowUi` for saving
on the same screen is unusual but architecturally sound — the two boundaries serve
distinct operations (load vs. composite save+upload). The work item documents this
explicitly. No action required.

## 4. Actionable Next Steps

### E1 — P3: `guardar` screens bypass boundary `getInitialState` (hardcoded heading)

**Finding**: Both `guardar.tsx:244` and `guardar/page.tsx:234` render a hardcoded heading
`"Guardar rascunho com imagem"`. The save-flow boundary's `getInitialState` returns
`title: 'Editar rascunho de ${petName}'` and `primaryAction: 'Guardar rascunho'`
(confirmed from `apps/mobile/src/pet-draft-save-flow.ts` line ~193). The SDD pattern
requires that ViewModel content — including copy — comes from the boundary, not from the
screen.

**Fix**: Load `petName` from the draft in `useEffect` (already available as `form.name`
after load). Show the save-flow boundary's heading after the draft loads. For the loading
state (before the draft is fetched), display the boundary's generic ready title via a
`useState` initializer that calls `saveFlowUi.getInitialState({ petName: null })` with a
mock client — exactly as `media.tsx` does for its boundary.

**Precedent**: `apps/mobile/app/animais/[petId]/media.tsx:35-41` — mock flow in
`useState` initializer to call `getInitialState` without a real auth client.

### E2 — P3: Missing load-error test cases for `guardar` screens

**Finding**: `tests/mobile/pet-draft-save-flow-screen.test.ts` and
`tests/web/pet-draft-save-flow-screen.test.ts` test only the `saveDraft` operation. The
`guardar` screen's loading path (`loadDraft` → `not_found` | `forbidden` | `failed`) is
not exercised by any test. The screen renders distinct UI for each of these three error
states (lines 161–185 in `guardar.tsx`).

**Fix**: Add 3 tests to each test file testing the load side via `createMobilePetDraftUi`:
- `loadDraft` → `not_found` → correct ViewModel state
- `loadDraft` → `forbidden` → correct ViewModel state
- `loadDraft` → `failed` → correct ViewModel state (with sanitization check on reasons)

**Precedent**: `tests/mobile/pet-draft-screen.test.ts` — tests `loadDraft` returning
`loaded`, `not_found`, `forbidden`, `failed` states for the edit screen.
