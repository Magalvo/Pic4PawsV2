## 3. Deviations and Scope Creep

None. Both work items are tightly scoped. The `requestUrl` addition to `PaymentWebhookVerifier`
is a deliberate forward-compatible interface change documented in the work item contract. The
`rawBody` parameter is retained so Eupago/Stripe adapters need no changes.

The method-enforcement logic in `routes/webhooks.ts` (405 for wrong method, provider-specific
Allow header) was not explicitly called out in the original IFTHENPAY-WEBHOOK-001 work item
spec but is directly required by the protocol change (GET vs. POST) and is well tested. This
is acceptable implementation detail, not scope creep.

### P2-2: Document or resolve the mobile `/abrigos` public route divergence

**File**: `apps/mobile/src/nav.ts` and/or a new work item

The web middleware (`apps/web/middleware.ts`) treats `/abrigos` and `/abrigos/[shelterId]`
as public (unauthenticated access allowed). The mobile `isPublicRoute` does not include them,
meaning unauthenticated mobile users are redirected to sign-in when navigating to the shelter
search tab.

If this is intentional (mobile requires login for shelter browsing): add a comment in
`isPublicRoute` explaining the deliberate divergence.

If this is a gap: add the two clauses to `isPublicRoute` and cover them in
`tests/mobile/auth-guard.test.ts`:

```ts
// (app)/(tabs)/abrigos root
if (segments[0] === '(app)' && segments[1] === '(tabs)' && segments[2] === 'abrigos' && !segments[3]) return true;
// abrigos/[shelterId] public detail
if (segments[0] === 'abrigos' && segments[1] && !segments[2]) return true;
```

Precedent: the matching web middleware clauses in `apps/web/middleware.ts`.
