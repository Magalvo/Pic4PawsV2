## Findings By Severity

### P1 — Carried Forward (Open)

#### P1-6 — Provider payment ID persistence failure still returns 201

**Severity:** P1 (unchanged from prior report)

**Evidence:** `apps/workers/src/donation.ts:75` still declares `setProviderPaymentId`
as optional (`?:`). At lines 376–381 the call uses optional chaining
(`.setProviderPaymentId?.()`), and the `.catch()` handler calls `failDonation` but
does NOT return a failure response — execution falls through to the 201 return at
line 383. The work-item doc `PAYMENT-PROVIDER-ID-PERSISTENCE-001` does not exist on
`main`. No merge commit for PR #291 appears in `git log --merges` since the baseline.

**Impact:** Two broken scenarios persist:
1. When `setProviderPaymentId` is absent, optional chaining skips the call silently,
   the donation remains in `pending_payment` without a provider ID, and 201 is returned.
2. When the call throws, the `.catch()` handler marks the donation `failed` via
   `failDonation`, then falls through and returns 201 anyway. The donor receives a
   valid payment reference for a record that is simultaneously marked failed. Incoming
   webhooks keyed on `providerPaymentId` cannot reconcile the payment.

**Required action:** Make `setProviderPaymentId` non-optional in `DonationRepository`,
add an explicit guard (`if (!donationRepository.setProviderPaymentId)`) returning 502,
wrap the call in a try/catch that returns 502 on failure, and gate the 201 return on
successful persistence. Create work-item doc with all required sections. Merge to main.

**Precedent:** The fix was prepared in branch `agent/PAYMENT-PROVIDER-ID-PERSISTENCE-001`
and described as PR #291, but that PR was never merged. The fix is ready; it only
needs to be re-opened and merged.

### P3 — New

#### P3-1 — Factory supabase selects unused `mb_way_phone` column

**Severity:** P3

**Evidence:** `apps/workers/src/payment-reference-factory-supabase.ts:28` selects
`mb_way_phone` from `shelter_payment_configs` as part of the column list. After PR #289
(P1-5), the payment method and donor MB WAY phone come from `PaymentReferenceInput`
fields `paymentMethod` and `mbWayPhone`. The value `row.mb_way_phone` is never
referenced in the factory body after the column is typed at line 11.

**Impact:** Superfluous data transferred on every reference creation call. Minor
performance and maintainability concern; the unused field is a signal that the
pre-P1-5 method-inference logic was not fully cleaned up.

**Recommendation:** Remove `mb_way_phone` from the select string and its type
declaration in `ShelterReferenceConfigRow`.
