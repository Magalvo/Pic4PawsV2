## Findings By Severity

### P0

No P0 finding was proven from the audited state. The P1 security and financial findings
below must nevertheless block production activation of automated payments.

### P1

#### P1-1 - Payment configuration table has no RLS protection

**Severity:** P1

**Evidence:** `packages/database/src/migration-artifacts.ts:362` creates
`public.shelter_payment_configs`; migration `0008` adds credential fields at lines
393-413, but neither migration enables RLS. `packages/database/src/rls-policies.ts:14-22`
omits this table from the policy matrix. The table contains IBAN, MB WAY phone, encrypted
Eupago credentials and a plaintext Ifthenpay anti-phishing key
(`packages/database/src/schema.ts:205-224`).

**Impact:** under normal Supabase public-schema grants, anonymous or authenticated REST
roles may be able to read or mutate financial configuration unless grants were revoked
manually outside the repository. Exposure of the plaintext anti-phishing key can enable
forged Ifthenpay callbacks; IBAN and phone exposure is also a GDPR/privacy incident.

**Recommendation:** add a non-destructive migration that enables RLS immediately, revoke
direct client access to credential columns, and define least-privilege owner/admin policies.
Prefer a server-only credential table or security-definer RPC that never returns secrets.
Rotate provider credentials after remediation if this migration reached any shared environment.

**Required test:** local Supabase policy tests proving anon, adopter and cross-tenant members
cannot select/update the row; only the correct owner/admin operation is allowed, and no client
response can retrieve credential columns.

#### P1-2 - Eupago webhook contract does not match the official protocol

**Severity:** P1

**Evidence:** `apps/workers/src/payment-webhook.ts:59-63` expects
`x-eupago-signature`; `apps/workers/src/eupago-verifier.ts:30-33` decodes a hexadecimal
signature; and lines 4-15/53 expect a flat `transactionId` payload with status `Success`.
Official Realtime Webhooks 2.0 use `X-Signature`, a base64-encoded HMAC, a nested
`transactions` object and statuses such as `Paid`, `Refund`, `Error`, `Cancel` and `Expired`.
The official legacy 1.0 callback is also different: URL/query parameters with Portuguese
field names and an API key.

**Impact:** real Eupago callbacks are rejected with 401 before the atomic transition, so a
paid donation remains `pending_payment` and cannot reconcile automatically.

**Recommendation:** select one documented Eupago webhook version explicitly, persist that
decision in the work item, and implement its exact header, encoding, raw-body and payload
contract. Confirm the provider transaction identifier used by both reference creation and
webhook lookup.

**Required test:** sanitized fixtures captured from the official sandbox for Multibanco and
MB WAY, including valid/invalid base64 signatures, `Paid`, retries and non-paid statuses.

#### P1-3 - Reference factory selects a database column that does not exist

**Severity:** P1

**Evidence:** `apps/workers/src/payment-reference-factory-supabase.ts:28` selects
`ifthenpay_api_key_encrypted`. That column is absent from the canonical schema
(`packages/database/src/schema.ts:205-224`) and from migration `0008`
(`packages/database/src/migration-artifacts.ts:393-413`). The committed columns are the
generic `api_key_encrypted`, Eupago-specific fields, and `ifthenpay_anti_phishing_key`.

**Impact:** PostgREST rejects the entire select before the provider branch runs. This blocks
Eupago as well as Ifthenpay reference creation, causing the newly created donation to be
failed and the donor to receive `502 payment_reference_failed`.

**Recommendation:** define one canonical typed credential schema and make config writes,
factory reads, Drizzle schema and migrations use exactly the same names. Avoid untyped string
selects for security-sensitive columns where generated database types can enforce drift.

**Required test:** integration test against a database built from committed migrations that
saves provider configuration and creates a reference factory for each provider.

#### P1-4 - Global Ifthenpay verifier can override the Eupago route

**Severity:** P1

**Evidence:** production composition sets `paymentWebhookVerifier` to the Ifthenpay verifier
when `PAYMENT_PRIMARY_PROVIDER=ifthenpay` (`apps/workers/src/index.ts:642-650`). The Eupago
route then prefers that injected verifier over `createEupagoVerifier()`
(`apps/workers/src/routes/webhooks.ts:120-125`). This contradicts the work-item decision that
provider-specific routes operate concurrently and no longer depend on the global primary provider.

**Impact:** any shelter configured for Eupago receives 401 callbacks whenever the deployment's
global primary provider is Ifthenpay.

**Recommendation:** remove the shared verifier dependency from provider-specific routes or
inject a verifier map keyed by provider. Route identity must select the verifier deterministically.

**Required test:** production-composition matrix for both values of `PAYMENT_PRIMARY_PROVIDER`,
proving valid Ifthenpay and Eupago callbacks succeed concurrently.

#### P1-5 - The selected payment method is lost before the PSP call

**Severity:** P1

**Evidence:** the donor sends `paymentMethod` and the database stores it
(`apps/workers/src/donation.ts:327-350`), but `PaymentReferenceInput` does not contain the
method. Both adapters instead infer MB WAY from the shelter's `mb_way_phone`
(`apps/workers/src/eupago-reference-adapter.ts:121-129` and
`apps/workers/src/ifthenpay-reference-adapter.ts:83-115`). Automated config writes always set
that phone to null (`apps/workers/src/shelter-payment-config.ts:291-317`), so the adapters
choose Multibanco. Web and Mobile default to MB WAY and also offer card/bank transfer
(`apps/web/app/abrigos/[shelterId]/doar/page.tsx:157-165` and
`apps/mobile/app/abrigos/[shelterId]/doar.tsx:43-56`).

**Impact:** MB WAY and card requests can produce a Multibanco reference while the transaction
is recorded under the user-selected method. This breaks user intent, accounting and provider
reconciliation. The flow also has no donor phone input for an MB WAY request.

**Recommendation:** carry a validated payment-method-specific union into the factory, including
the donor's MB WAY phone where required. Derive allowed methods from the shelter's active tier
and provider; only render methods the server reports as configured.

**Required test:** end-to-end contract cases for each advertised method asserting request payload,
provider endpoint, returned reference type and persisted `payment_method` all agree.

#### P1-6 - Provider payment ID persistence failure still returns success

**Severity:** P1

**Evidence:** after obtaining a PSP reference, `apps/workers/src/donation.ts:357-371` catches a
`setProviderPaymentId` failure, optionally marks the donation failed, and still returns
`201 donation_created` with the usable reference. The repository operation is optional in the
interface (`apps/workers/src/donation.ts:70-77`).

**Impact:** the donor can complete a real payment that the webhook cannot associate with the
local donation. Funds may settle while the local record is failed or retains a placeholder ID.

**Recommendation:** make provider-ID persistence mandatory and part of a recoverable state
machine. Do not return the reference as successful until its provider identifier is durably
stored; add an outbox/reconciliation path for the external-call/database-write split.

**Required test:** force the provider-ID update to fail and assert no 201 is returned; add a
race test where a webhook arrives immediately after reference creation.

#### P1-7 - Provider switch guard can be bypassed by switching to manual

**Severity:** P1

**Evidence:** the pending-donation guard runs only when the new payload is `automated`
(`apps/workers/src/shelter-payment-config.ts:267-277`). The manual branch then clears
`activeProvider` and every provider credential at lines 281-290.

**Impact:** an owner can switch to manual while automated donations are pending. Subsequent
callbacks can no longer resolve the old per-shelter secret and are rejected, stranding payments.

**Recommendation:** guard every transition or credential rotation that can invalidate pending
payments. Retain versioned old credentials until all associated transactions settle.

**Required test:** pending payment plus automated-to-manual switch must return 409; same-provider
secret rotation must either be blocked or preserve the previous verification key safely.

#### P1-8 - Mobile donation screen creates an unauthenticated Supabase client

**Severity:** P1

**Evidence:** `apps/mobile/app/abrigos/[shelterId]/doar.tsx:74-80` creates a new Supabase client
inside submit with `persistSession: false`, then asks that new client for its session. It does not
use the authenticated `mobileSupabaseClient` singleton established by the navigation/auth work.

**Impact:** the access token is normally null, so real mobile donation submission is
unauthenticated even though boundary-only tests pass.

**Recommendation:** use the shared mobile singleton and construct the donation client once at
the screen boundary. Keep submit exception handling in `try/finally` so failures cannot leave the
form permanently disabled.

**Required test:** screen-level integration test with the shared signed-in session proving the
Worker receives the bearer token and the unauthenticated case remains explicit.

### P2

#### P2-1 - Encryption key contract is not enforced

**Severity:** P2

**Evidence:** the work item requires at least 32 bytes, but `packages/config/src/env.ts:41`
uses the generic optional secret validator with no minimum. `apps/workers/src/crypto.ts:9-11`
truncates the first 32 JavaScript characters before UTF-8 encoding, which can produce an invalid
AES key length for non-ASCII input. The Ifthenpay anti-phishing key is stored plaintext by the
current schema and save path.

**Impact:** malformed deployment secrets fail only at runtime; short but valid AES keys weaken
the intended operational contract, and the plaintext webhook credential magnifies P1-1.

**Recommendation:** require an explicit 32-byte base64/hex key format at environment parsing,
decode it deterministically, and encrypt every provider secret consistently. Document rotation
and key-version handling.

**Required test:** reject short, malformed and non-ASCII ambiguous keys at startup; round-trip
with a decoded 256-bit key and verify all provider credentials are ciphertext at rest.

#### P2-2 - Graphify architecture map is stale

**Severity:** P2

**Evidence:** `graphify-out/GRAPH_REPORT.md` was built from `29b764b4`; audited `main` is
`1a9e8b3`. It therefore does not represent the Eupago DB, verifier, factory, adapters or client
boundaries introduced by PRs #277-#281.

**Impact:** the mandatory architecture map omits the highest-risk new payment path and can mislead
future audits or implementation work.

**Recommendation:** run `graphify update .` in the implementation branch after the code fixes and
commit the refreshed report/output according to repository policy.

**Required test:** CI check comparing the Graph Report freshness commit to `HEAD` or to the latest
code-changing commit.

#### P2-3 - TDD tests reproduce an unverified provider contract

**Severity:** P2

**Evidence:** `tests/workers/eupago-verifier.test.ts:22-49` creates flat `Success` payloads and
hex signatures that mirror the implementation rather than official provider fixtures.
`tests/workers/eupago-reference-adapter.test.ts:24-126` selects the method from a shelter phone.
`tests/workers/donation.test.ts:424-462` submits the default MB WAY payload but expects a
Multibanco reference. All tests pass while the user/provider contract is inconsistent.

**Impact:** strict TDD gives false confidence because the oracle is the local assumption, not the
external system. The checkpoint and `agent-resume.md` consequently overstate the track as complete.

**Recommendation:** amend the work items with source/version/date for every PSP contract, add
sanitized official fixtures, and require a sandbox contract test before marking payment-provider
work `done`.

**Required test:** provider contract suite generated from official examples/OpenAPI plus one
recorded sandbox smoke flow for reference creation and webhook confirmation.

## Executive Summary

The repository has strong internal structure and green quality gates, but the newly declared
Eupago track is not production-ready. The current code has multiple independent blockers: the
credential table lacks RLS, the reference factory queries a nonexistent column, the webhook
protocol differs from current official documentation, verifier composition is not truly
multi-provider, and the selected payment method is not propagated to the PSP. Mobile donation
submission also bypasses the authenticated singleton.

The principal lesson is not a failure of test quantity; it is oracle quality. The 2515 passing
tests prove the code matches its local assumptions. They do not yet prove that those assumptions
match Supabase policy behavior or the Eupago protocol.

## Residual Risks

- No deployed Supabase grants were inspected; P1-1 may be mitigated by undocumented external
  revocations, but the repository does not establish or test them.

- A full secret scan was intentionally not performed because the audit rules prohibit global
  repository scans; only selected payment/configuration files were inspected.
