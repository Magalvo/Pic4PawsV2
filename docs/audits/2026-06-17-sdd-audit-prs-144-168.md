## 3. Deviations and Scope Creep

No scope creep observed. All changes are traceable to their work items.

One architectural deviation: the WEB-AUTH-PAGE-001 spec requires `persistSession: false` but subsequent pages (e.g., WEB-SHELTER-UPDATE-PAGE-001) rely on `supabase.auth.getSession()` to retrieve the stored session. Using `persistSession: false` would break cross-page auth flows. The implementation chose `persistSession: true` (default), which is architecturally correct but was not reflected back into the spec.

### E2 — P2: WEB-AUTH-PAGE-001 spec requires `persistSession: false` — implementation uses default `persistSession: true`

**Files**:
- `apps/web/app/entrar/page.tsx:13`
- `apps/web/src/supabase-browser.ts`

The WEB-AUTH-PAGE-001 work item spec states:

> `persistSession: false` — no localStorage writes; token lives in React state only

The implementation creates the Supabase client without this option:

```typescript
// apps/web/app/entrar/page.tsx
const authClient = createClient(supabaseUrl(), supabaseAnonKey());
// No { auth: { persistSession: false } } option

// apps/web/src/supabase-browser.ts
export const createSupabaseBrowserClient = () =>
  createClient(supabaseUrl(), supabaseAnonKey());
```

The default is `persistSession: true`, which writes tokens to `localStorage`. The deviation is architecturally justified: the shelter update page (`/abrigos/[shelterId]/editar`) reads the session via `supabase.auth.getSession()` and requires a persisted session to obtain the access token for authenticated requests. Without `persistSession: true`, users who navigate from `/entrar` to a protected page would have no active session.

However, the spec has not been updated to document this decision, and no test validates the Supabase client options.

**Fix**:
1. Update `docs/work-items/WEB-AUTH-PAGE-001-web-auth-page.md` to remove the `persistSession: false` requirement and document that `persistSession: true` is required for cross-page token propagation via `getSession()`.
2. Add a `Completion Notes` section to WEB-AUTH-PAGE-001 explaining the architectural decision.
3. Optionally add a test that confirms `getSession()` returns the session after `signIn` (integration-level).

### E4 — P3: MOBILE-PET-FEED-SCREEN-001 `loaded` state not present in implementation

**File**: `apps/mobile/app/animais/index.tsx`

The MOBILE-PET-FEED-SCREEN-001 work item defines five states including `loaded` ("list of pets rendered"). The implementation renders the loaded state as the final `else`/default branch without an explicit `viewModel.state === 'loaded'` guard:

```typescript
// No explicit `if (viewModel.state === 'loaded')` check
return (
  <SafeAreaView>
    <ScrollView>
      {viewModel.pets.map(...)}
    </ScrollView>
  </SafeAreaView>
);
```

If the `createMobilePetFeedUi` boundary returns an unexpected state (e.g., `idle` with pets present), the screen would render the pets list without checking state. This mirrors an identical pattern in the web pet feed page. The web page also handles `loaded` implicitly. While the boundary test suite likely prevents unexpected states, the guard makes the intent explicit and future-proofs against state additions.

This is consistent across all public screen implementations (web and mobile), so it appears to be an intentional pattern. However, it diverges from the explicit state-branch pattern used in all prior mobile boundaries.

**Fix**: Add explicit `if (viewModel.state === 'loaded')` guard before rendering the pets/shelters list in mobile screen components. This is low-risk, purely defensive.
