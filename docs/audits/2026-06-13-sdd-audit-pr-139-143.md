## 4. Findings

### E1 — P2: `shelter-delete-supabase.ts` leaks raw Supabase error messages

**File**: `apps/workers/src/shelter-delete-supabase.ts:36`

```typescript
throw new SupabaseShelterDeletionRepositoryError(
  `Failed to delete shelter: ${result.error.message ?? 'unknown error'}`,
);
```

Every other repository adapter in the codebase uses a fixed sanitized string and never
interpolates `result.error.message`. Examples:

- `assertShelterResult` in `shelter-supabase.ts` throws `'Failed to load shelter profile'`
- `assertSupabaseResult` in `pet-supabase.ts` throws the fixed message string

If Supabase returns an error whose message contains a bearer token, service-role credential, or
other credential string (e.g., from an improperly configured client or from a debug trace in
the error path), that credential will appear in the thrown exception. While the Worker error
boundary (WORKER-ERROR-BOUNDARY-001) prevents the raw exception from reaching HTTP clients,
it will appear in any log or trace that captures uncaught exceptions.

**Additionally**: `tests/workers/shelter-delete-supabase.test.ts` has no sanitization test
asserting that the `service-role` and `bearer` marker substrings are absent from the thrown
exception. This is the required test convention for every adapter that throws on Supabase
errors.

**Fix**:

1. Replace the interpolated message with a fixed string:

   ```typescript
   if (result.error) {
     throw new SupabaseShelterDeletionRepositoryError('Failed to delete shelter');
   }
   ```

2. Add a sanitization test to `tests/workers/shelter-delete-supabase.test.ts`:

   ```typescript
   it('throws sanitized error without leaking secrets on Supabase errors', async () => {
     const client = makeClient({
       data: null,
       error: { message: 'service-role key: abc bearer xyz' },
     });
     const { shelterDeletionRepository } = createSupabaseShelterDeletionRepositories({ client });

     const promise = shelterDeletionRepository.deleteShelter('shelter-a');
     await expect(promise).rejects.toBeInstanceOf(SupabaseShelterDeletionRepositoryError);

     const err = await promise.catch((e: unknown) => e as Error);
     expect(err.message).not.toMatch(/service-role|bearer/i);
   });
   ```

### E2 — P3: Sanitization test regex missing explicit `bearer` pattern check

**Files**:
- `tests/workers/pet-supabase-repository.test.ts:615`
- `tests/workers/shelter-supabase-repository.test.ts:230`

Both tests assert:

```typescript
rejects.not.toThrow(/service-role-secret|r2-secret-key|test-token/)
```

The house convention (from SDD audit skill) requires that sanitization tests assert BOTH
`service-role` AND `bearer` pattern substrings are absent. The current regex does not include
`bearer` as a distinct pattern. `test-token` (the value after `Bearer`) is checked, but the
`bearer` keyword itself is not.

This means if only the word `Bearer` (without a following token) appeared in an error message,
it would not be caught by the current test.

**Fix**: Update the regex in both files to include the `bearer` pattern:

```typescript
rejects.not.toThrow(/service-role|bearer/i)
```

Or extend the existing regex:

```typescript
rejects.not.toThrow(/service-role-secret|r2-secret-key|test-token|bearer/i)
```
