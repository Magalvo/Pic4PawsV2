## 1. Score: 6/10

The architecture, auth invariants, and domain security rules are correctly implemented across
all eight PRs: donor identity is always derived from `actor.id` (never the request body),
shelter membership uses `canManageShelter`, `paid_at`/`reviewed_at` are server-clock only,
and `notifyDonationPaid` is correctly fire-and-forget. The SDD spec compliance and test
coverage are strong.

Four points deducted across three severity tiers:

- **−2 Critical**: The mobile receipt upload feature is non-functional (mock file), and all
  three Supabase state-transition updates silently return HTTP 200 on concurrent no-ops.
- **−1 High**: Wrong `purpose` enum on receipt media uploads (both platforms); mobile payment
  config screen re-creates all clients on every submit.
- **−1 Medium/Low**: New Supabase client per mount in the review screen; non-null assertion
  crash risk in Alert callbacks; cross-asset media ownership; `now()` called at dispatch
  time; duplicated helpers; no double-submit guard; soft-deleted row mutation gap.

## 3. Findings

### F-01 — CRITICAL: Supabase .update() without rows-affected check — false HTTP 200 on concurrent state change

**Files**: `apps/workers/src/donation-manual-supabase.ts` lines 85, 99, 118
**Also**: `apps/workers/src/donation-manual.ts` lines 141, 207

`submitReceipt`, `approveDonation`, and `rejectDonation` all call `.update()...eq('status', <expected>)` without chaining `.select()`. Supabase returns `{ data: null, error: null }` regardless of whether 0 or N rows were affected. Only `result.error` is checked.

**Failure scenario**: Two concurrent PATCH `/receipt` or `/review` requests both pass the in-process status guard (TOCTOU gap between `getDonationForAction` and the update). The second request's WHERE clause matches 0 rows but returns no error. The handler returns HTTP 200 with `receipt_submitted` / `donation_approved` / `donation_rejected` — a false success. For the approval path, `notifyDonationPaid` is also fired for a transition that never landed in the DB.

**Fix**: Chain `.select('id')` on each update. Check `result.data?.length === 0` → return `donation_wrong_state` (409) instead of the success body.

### F-02 — CRITICAL: Mobile receipt upload uses hardcoded placeholder — feature is non-functional

**File**: `apps/mobile/app/doacoes/[donationId]/comprovativo.tsx` line 84

`handleSelectAndUpload` constructs a mock `MobileDonationReceiptFileInput` with `uri: 'file:///data/placeholder/receipt.jpg'` and `size: 0`. `expo-image-picker` and `expo-document-picker` are both absent from `apps/mobile/package.json`. Every receipt upload sends this placeholder; no real file is ever selected or transmitted.

**Fix**: Add `expo-image-picker` (or `expo-document-picker`) and replace the mock object with a real picker call. Ensure the `onPress` async result is awaited or its rejection is caught.

---

### F-03 — HIGH: Wrong `purpose: 'identity_document'` on donation receipt media uploads

**Files**: `apps/mobile/src/donation-receipt.ts` line 209, `apps/web/src/donation-receipt.ts` line 195

Both UI modules create the media upload intent with `purpose: 'identity_document'`, cargo-culted from the identity document upload flow. Donation receipts are filed under the identity-document storage path/bucket in any purpose-aware storage policy or CDN routing rule.

**Fix**: Replace `purpose: 'identity_document'` with the appropriate value (e.g. `'donation_receipt'`) in both files. Confirm the worker's media-upload handler accepts this purpose value.

---

### F-04 — HIGH: Mobile payment config `handleSubmit` re-creates all clients on every press — no `useRef`, double-submit race

**File**: `apps/mobile/app/abrigos/[shelterId]/pagamento.tsx` lines 67–88

`handleSubmit` creates fresh `createSavePaymentConfigClient`, `createLoadPaymentConfigClient`, and `createMobileShelterPaymentConfigUi` instances on every invocation. There is no `useRef`. Unlike every other mobile screen in the codebase (e.g. `comprovativo.tsx`, `[donationId].tsx`), the UI instance from `useEffect` is discarded and a new one is created per submit. Two rapid presses launch two independent `saveConfig` calls with no concurrency guard; whichever resolves last sets the final `viewModel`.

**Fix**: Add `useRef<ReturnType<typeof createMobileShelterPaymentConfigUi> | null>(null)`, store the UI in `useEffect`, and reuse it in `handleSubmit`. Add a `submitting` guard around the button press.

---

### F-05 — HIGH: Async `Alert.alert` `onPress` not awaited — unhandled rejection freezes UI in uploading state

**File**: `apps/mobile/app/doacoes/[donationId]/comprovativo.tsx` line 73

`Alert.alert` does not await the `async onPress` callback. If `uploadAndSubmit` throws, the rejection is unhandled. `setViewModel` is never called with an error state, leaving the screen permanently stuck showing "A processar..." with no user-visible error and no retry path.

**Same pattern** exists in `apps/mobile/app/abrigos/[shelterId]/doacoes/[donationId].tsx` Alert callbacks (handleApprove/handleReject).

**Fix**: Wrap the `onPress` body in `try/catch` and call `setViewModel` with a `failed` state in the catch branch.

---

### F-06 — MEDIUM: Mobile donation review screen calls `createClient()` on every mount — leaks Supabase connections

**File**: `apps/mobile/app/abrigos/[shelterId]/doacoes/[donationId].tsx` lines 20–21, 31

`import { createClient } from '@supabase/supabase-js'` and `createClient(supabaseUrl(), supabaseAnonKey())` are called inside `useEffect`, creating a new Supabase client (with its own auth state machine and realtime WebSocket) on each screen mount. The old client is never disposed. Every other mobile screen that needs auth uses `mobileSupabaseClient` (the singleton from `apps/mobile/src/supabase.ts`).

**Fix**: Remove the raw `createClient` import and use `mobileSupabaseClient` as in `comprovativo.tsx` line 22.

---

### F-07 — MEDIUM: `uiRef.current!` non-null assertion inside async `Alert.alert` callback — TypeError crash on component unmount

**File**: `apps/mobile/app/abrigos/[shelterId]/doacoes/[donationId].tsx` line 64

`handleApprove` shows a non-blocking Alert then returns. If the component unmounts before the user taps "Aprovar" (navigation race), `uiRef.current` is `null`. The async `onPress` then executes `uiRef.current!.approve(donationId)`, throws `TypeError: Cannot read properties of null`, and crashes the JS thread. `handleReject` has the same pattern.

**Fix**: Guard inside the async callback: `if (!uiRef.current) return;` before the non-null assertion.

---

### F-08 — MEDIUM: `verifyMediaOwnership` allows any of the donor's media assets to be attached as receipt

**File**: `apps/workers/src/donation-manual-supabase.ts` line 67

The ownership check filters only by `created_by_user_id`. A donor who knows the UUID of any media asset they previously uploaded (a pet photo, a shelter banner) can pass that UUID as `receiptMediaId`. The check passes and an unrelated asset is attached as a receipt, which the shelter manager then reviews as the "comprovativo".

**Fix**: Add a `purpose` filter to the query (`.eq('purpose', 'donation_receipt')` or equivalent) so only assets uploaded with the correct intent can be used as receipts. This requires the purpose bug in F-03 to be fixed first.

---

### F-09 — MEDIUM: `now()` called at route-dispatch time — `reviewedAt`/`paidAt` timestamps predate the actual DB write

**File**: `apps/workers/src/routes/donations.ts` line 45

```typescript
now: dependencies.now?.() ?? new Date().toISOString()
```

The timestamp is captured before `handleReviewDonationRequest` does any auth, validation, or media-ownership work. If any of those async steps take time, `reviewed_at` / `paid_at` in the DB will be earlier than the actual moment the row was written — systematically undercounting processing time in audit logs. In test doubles that inject a controllable `now` factory, the factory is called once per dispatch, not once per write.

**Fix**: Pass `now: dependencies.now ?? (() => new Date().toISOString())` (the factory, not its result) and evaluate it inside `handleReviewDonationRequest` immediately before the `approveDonation`/`rejectDonation` call.

---

### F-10 — LOW: No empty-string guard on `uploadResult.mediaId` after successful upload

**File**: `apps/mobile/src/donation-receipt.ts` line 233 (same logic in web counterpart)

If the upload worker returns `{ ok: true, status: 'uploaded', mediaId: '' }`, the failure branch is skipped and `submitReceipt` is called with `receiptMediaId: ''`. The worker's `isNonEmptyString` check rejects it (400, `receipt_media_id_required`), which the client maps to `'worker_request_failed'` via the failure-status fallback. The user sees "Não foi possível enviar" after the upload appeared to succeed, with no actionable cause.

**Fix**: After the upload success branch, add `if (!uploadResult.mediaId?.trim()) return { state: 'failed', ... }` before calling `submitReceipt`.

---

### F-11 — LOW: No double-submit guard on the upload button in `comprovativo.tsx`

**File**: `apps/mobile/app/doacoes/[donationId]/comprovativo.tsx` around line 78

The "Selecionar comprovativo" `TouchableOpacity` has no `disabled` prop tied to an uploading state. Two rapid taps produce two concurrent `uploadAndSubmit` calls, each attaching a potentially different `mediaId` to the donation.

**Fix**: Add a `submitting` state and set `disabled={submitting}` on the button before showing the Alert.

---

### F-12 — LOW: Supabase update methods have no `.is('deleted_at', null)` filter — soft-deleted donations can be mutated

**File**: `apps/workers/src/donation-manual-supabase.ts` lines 86–120

`getDonationForAction` correctly filters `.is('deleted_at', null)`. But the three update methods (`submitReceipt`, `approveDonation`, `rejectDonation`) have no such filter. If a donation is soft-deleted between the read and the update (T1 → T2), the update still modifies the soft-deleted row.

**Fix**: Add `.is('deleted_at', null)` to the WHERE chain in each update method.

---

### F-13 — LOW: `jsonResponse` and `extractBearerToken` duplicated in `donation-manual.ts` instead of imported from `routes/shared.ts`

**File**: `apps/workers/src/donation-manual.ts` lines 72, 75

`routes/shared.ts` already exports `jsonResponse` (line 9) and `parseAuthorizationBearer` (line 20), which are semantically identical to the private copies in `donation-manual.ts`. If `shared.ts` ever changes its Bearer-parsing logic (e.g. case-insensitive `Authorization` header for RFC 7235 compliance), `donation-manual.ts` silently diverges.

**Fix**: Delete the local copies and import from `'./routes/shared'`.

---

### F-14 — LOW: `handleSubmit` guard in `pagamento.tsx` permits execution from non-idle `viewModel` states

**File**: `apps/mobile/app/abrigos/[shelterId]/pagamento.tsx` line 63

The guard `if (submitting || !viewModel) return` permits `saveConfig` to run when `viewModel.state` is `'failed'` or `'forbidden'`. While not normally reachable through the UI, an accessibility tool or programmatic trigger would produce confusing behaviour: `saveConfig` runs, returns a new state, and the UI may flicker since the `'idle'` form branch is the only one that renders a form.

**Fix**: Change the guard to `if (submitting || !viewModel || viewModel.state !== 'idle') return`.

---

## 4. Recommendations

1. **Fix F-01 immediately before next production deploy**: The false-HTTP-200 on concurrent state changes is a correctness and audit-log integrity issue. Add `.select('id')` + rows-affected check to all three Supabase update methods.

2. **Fix F-02 before any user-facing receipt upload**: The mobile receipt feature does not work. Wire a real file picker before the screen is accessible to donors.

3. **Fix F-03 (wrong purpose) alongside F-02**: The correct purpose value must be in place before real uploads start, so receipts are not mis-filed under identity-document storage paths.

4. **F-04, F-06, F-07 are one-line fixes**: The `useRef` pattern for mobile payment config, the singleton Supabase client, and the null-guard inside Alert callbacks are low-effort and high-value.

5. **F-08 depends on F-03**: Once `purpose: 'donation_receipt'` is established, `verifyMediaOwnership` can scope by purpose to prevent cross-asset-type attachment.
