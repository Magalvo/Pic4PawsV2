## 4. Actionable Next Steps

**P2-1 — Document two-step registration atomicity gap (new work item)**

`apps/workers/src/user-register-supabase.ts`: if `auth.admin.createUser()` succeeds but
the `register_user` RPC subsequently fails (e.g. duplicate `auth_user_id`, DB constraint),
the Supabase repository throws `SupabaseUserRegistrationRepositoryError`. The worker's
global error handler returns a 500; the client returns `worker_request_failed`. On retry,
`auth.admin.createUser()` will now return "already registered", causing the client to
surface `email_already_registered` — stranding the user with an orphaned auth account and
no public profile row.

Recommended remediation: after `createUser` succeeds, if the RPC fails, attempt
`auth.admin.deleteUser(authUserId)` to roll back the auth user before re-throwing.
Create work item `USER-REGISTER-ROLLBACK-001`. Precedent: the two-step pattern in
`apps/workers/src/user-register-supabase.ts` itself.

**P2-2 — Stabilize email-already-registered detection (risk, low urgency)**

`apps/workers/src/user-register-supabase.ts:isEmailAlreadyRegistered`: detection relies
on Supabase error message string matching (`'already registered'`,
`'already been registered'`). Supabase does not guarantee error message stability across
versions. If the message changes, the function falls through to a thrown
`SupabaseUserRegistrationRepositoryError` (500) rather than a 409.

Recommended: also check the Postgres error code (e.g. `error.code === '23505'` for
unique violation) as a secondary signal, or consult the Supabase admin API error type
field. No immediate work item needed — flag for when Supabase SDK is upgraded.
