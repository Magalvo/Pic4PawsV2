## 1. Spec Compliance Score: 9/10

PRs #209–#212 close all prior P2/P3 findings cleanly (PR #210) and deliver the shelter
verification workflow (worker + client + web + mobile) as specified across three work items
(SHELTER-VERIFY-001, SHELTER-VERIFY-WEB-001, SHELTER-VERIFY-MOBILE-001). The implementation
is protocol-correct: transition guard, role split (canManageShelter vs canVerifyShelter),
route ordering (verification before profile), Supabase soft-delete filter, and PT-PT copy
all match the spec. All four validation gates pass cleanly.

One point deducted for two convention gaps:

- The sanitization tests in `tests/web/shelter-verify-ui.test.ts` and
  `tests/mobile/shelter-verify-ui.test.ts` only cover the generic fallback branch. The
  specific error branches (forbidden, invalid_transition, shelter_not_found, unauthenticated)
  pass `result.reasons` directly without UI-layer sanitization and have no test asserting
  the serialized state is safe for those paths. Runtime risk is low (the client layer already
  sanitizes), but the convention requires asserting both `service-role` and `bearer` absent
  from the serialized state for every failed path.

- The route-table test (`tests/workers/route-table.test.ts`) has no entry for
  `matchWorkerShelterVerificationId`. The route ordering guard test exists specifically to
  prevent regression on ordering bugs, and the verification route's critical position (before
  profile) is not covered by it.

No security or correctness issues were found.

requires a "Completion Notes" section header, and both web and mobile work items satisfy the

button in both web and mobile UIs. The spec says "reinstate" is supported; there is no dedicated
UI button with the label "Reinstalar" — the admin reuses "Verificar abrigo" for both initial
verification and reinstatement. This is an acceptable UX design choice, and server-side
transition guards enforce correctness regardless.

### P2-1: Add sanitization assertions for specific error branches (web + mobile)

**Files**:
- `tests/web/shelter-verify-ui.test.ts`
- `tests/mobile/shelter-verify-ui.test.ts`

The specific error branches (forbidden, invalid_transition, shelter_not_found, unauthenticated)
in both `createWebShelterVerifyUi` and `createMobileShelterVerifyUi` pass `result.reasons`
directly without UI-layer sanitization. The client layer sanitizes first, so runtime risk is
low, but the house convention requires asserting that service-role and bearer are absent from
the serialized state for every failed path, not just the generic fallback.

Add one test per file that poisons the reasons on a specific branch (e.g. forbidden) and
asserts the serialized state does not contain service-role or bearer:

```ts
it('specific error branches do not expose credentials in serialized state', async () => {
  const ui = createWebShelterVerifyUi({
    shelterVerificationClient: makeClient({
      ok: false,
      status: 'forbidden',
      reasons: ['forbidden', 'service-role-key', 'bearer eyJ...'],
    }),
  });
  const state = await ui.updateVerificationStatus('shelter-1', 'verified');
  const serialized = JSON.stringify(state).toLowerCase();
  expect(serialized).not.toContain('service-role');
  expect(serialized).not.toContain('bearer ');
});
```

Note: this will currently FAIL for the specific branches because they pass reasons directly.
The fix is to apply sanitizeReasons on all branches, not just the generic fallback. This is
a defense-in-depth change.

Precedent: `tests/web/auth-page.test.ts:69-83` and `tests/mobile/auth-screen.test.ts:89-99`.
